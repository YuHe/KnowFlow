## 1. 产品概述
### 1.1 产品定位
KnowFlow 是一个**在线知识管理平台**，面向团队提供文档撰写、知识沉淀、权限管理和协作分享的一站式解决方案。

### 1.2 目标用户
* **系统管理员**：管理平台所有知识库和用户，拥有全局最高权限
* **团队管理者**：建立团队知识体系，管理成员权限
* **知识贡献者**：撰写、编辑、维护团队文档
* **知识消费者**：查阅、搜索、引用已有知识
* **外部访问者**：通过公开链接阅读公开文档（无需登录）

### 1.3 核心价值
|维度|说明|
|-|-|
|结构化知识管理|知识库 → 目录树 → 文档 → 卡片，四级组织体系|
|所见即所得编辑|富文本编辑器，支持 Markdown 快捷语法，降低写作门槛|
|细粒度权限控制|系统级 + 知识库级双重权限体系，覆盖全部操作场景|
|多格式导出|支持导出为 Markdown、Word（.docx）、PDF 三种格式|
|轻量私有部署|Docker 一键部署，数据自主可控|

### 1.4 产品边界
**V1.0 包含：**

|模块|说明|
|-|-|
|知识库管理|创建、编辑、删除知识库；设置名称、描述、图标、可见性|
|目录系统|多级嵌套目录树，支持拖拽排序、折叠展开、右键操作|
|文档编辑|富文本所见即所得编辑，支持 Markdown 快捷语法，自动保存|
|模板系统|系统内置模板 + 用户自定义模板，新建文档时可选|
|卡片拆分|文档内按分隔符拆分为独立卡片，可拖拽排序、独立编辑|
|图片/附件上传|支持行内图片粘贴/拖拽上传、附件下载链接|
|历史版本|每次手动保存（Ctrl+S）生成版本快照，支持预览和恢复|
|评论系统|文档级评论面板，支持添加、查看、删除评论|
|协作功能|知识库内多人可编辑（非实时协同，基于后写覆盖策略）|
|收藏功能|用户可收藏文档，在首页快速入口查看|
|全文搜索|知识库内/全局搜索，标题 + 正文命中，结果高亮|
|文档导出|支持导出为 Markdown（.md）、Word（.docx）、PDF 三种格式|
|文档分享|生成分享链接，支持设置访问权限和有效期|
|Markdown 导入|支持单文件 .md 导入、批量 .zip 导入（含目录结构解析）|
|对外公开访问|知识库或单文档可设为公开，匿名用户只读访问|
|Docker 部署|提供 docker-compose.yml，一键启动全部服务|

**V1.0 不包含（明确排除）：**

* 实时多人协同编辑（CRDT / OT）
* AI 智能写作 / 自动摘要
* 向量语义搜索
* 跨知识库文档引用网络
* 工作流 / 审批流
* 移动端 App（仅支持响应式 Web 访问）
* 第三方应用集成

## 2. 用户与角色模型
### 2.1 用户模型
|字段|类型|说明|
|-|-|-|
|id|UUID|主键|
|username|VARCHAR(64)|用户名，唯一，3-64 字符|
|display_name|VARCHAR(128)|显示名称|
|email|VARCHAR(255)|邮箱，唯一|
|password_hash|TEXT|bcrypt 加密的密码哈希|
|avatar_url|TEXT|头像 URL，可为空|
|created_at|TIMESTAMP|创建时间|
|updated_at|TIMESTAMP|最后更新时间|

### 2.2 系统角色定义
系统角色为全局角色，与知识库无关：

|角色|标识|权限概述|
|-|-|-|
|SuperAdmin|super_admin|系统管理员。可管理所有知识库、所有用户，拥有全局最高权限（不直接具有知识库内文档操作权，而是通过成员身份介入）。通过 `users` 表中 `role` 字段区分|
|User|user|普通用户。根据知识库成员身份获得对应权限|

**系统管理员能力：**

* 查看系统中所有知识库列表（包括非公开知识库）
* 查看所有用户列表
* 重置用户密码
* 启用/禁用用户账号
* 删除任意知识库
* 转移任意知识库的所有权
* 查看平台运营统计（知识库数、用户数、文档数、活跃度等）

**系统管理员不拥有的能力（明确排除）：**

* 不能绕过知识库成员身份直接查看非自己成员知识库中的文档内容（如需查看文档，需先将自身添加为成员）
* 不能修改用户创建的具体文档内容

### 2.3 知识库角色定义
角色在知识库级别绑定，一个用户在不同知识库中可以拥有不同角色。

|角色|标识|权限概述|
|-|-|-|
|Owner|owner|知识库创建者。拥有全部权限，不可被移除、不可被降级|
|Admin|admin|管理员。可管理成员、修改设置、管理所有文档。不可删除知识库|
|Editor|editor|编辑者。可创建/编辑/删除自己创建的文档，不可管理成员|
|Viewer|viewer|只读访问。仅可查看文档和评论，不可编辑|

### 2.4 权限能力矩阵
|操作|Owner|Admin|Editor|Viewer|
|-|-|-|-|-|
|查看文档/目录|✅|✅|✅|✅|
|查看评论|✅|✅|✅|✅|
|添加评论|✅|✅|✅|❌|
|删除自己的评论|✅|✅|✅|❌|
|删除他人的评论|✅|✅|❌|❌|
|创建文档|✅|✅|✅|❌|
|编辑文档|✅|✅|✅（仅自己创建）|❌|
|删除文档|✅|✅|✅（仅自己创建）|❌|
|创建/编辑/删除目录|✅|✅|✅|❌|
|上传附件/图片|✅|✅|✅|❌|
|收藏文档|✅|✅|✅|✅|
|查看历史版本|✅|✅|✅|❌|
|恢复历史版本|✅|✅|✅|❌|
|导出文档|✅|✅|✅|❌|
|分享文档|✅|✅|✅|❌|
|管理成员|✅|✅|❌|❌|
|修改知识库设置|✅|✅|❌|❌|
|设置公开可见|✅|✅|❌|❌|
|删除知识库|✅|❌|❌|❌|

### 2.5 可见性模型
|层级|可见性类型|说明|
|-|-|-|
|知识库|private|仅成员可访问，需要登录|
|知识库|public|任何人可只读访问，无需登录|
|文档|继承知识库|默认跟随知识库可见性|
|文档|单独设为公开|仅此文档公开，知识库其余内容不暴露|

## 3. 功能需求
### 3.1 知识库管理
#### 3.1.1 知识库列表页
**页面布局：**

* 顶部：搜索框 + 用户头像
* 内容区分两个 Tab：
    * Tab 1：「我的知识库」— 用户拥有/管理的知识库
    * Tab 2：「参与的知识库」— 用户以 Viewer/Editor 身份加入的知识库

* 每个知识库卡片显示：图标（emoji）、名称、描述（两行截断）、成员数、文档数、最近更新时间
* 卡片右上角显示用户角色标签（拥有者/管理员/编辑者/阅读者）
* 支持按更新时间/名称排序

**操作入口：**

* 页面右上角「新建知识库」按钮
* 知识库卡片点击进入知识库首页
* 知识库卡片右键菜单：编辑设置、删除（仅 Owner 可见）

#### 3.1.2 新建/编辑知识库
**新建弹窗字段：**

|字段|类型|必填|说明|
|-|-|-|-|
|名称|文本输入|✅|1-128 字符，知识库内唯一（在用户创建的知识库中）|
|描述|多行文本|❌|知识库简介，最多 500 字符|
|图标|emoji 选择器|❌|默认 📁，支持从 200+ emoji 中选择|
|可见性|单选|✅|`private`（仅成员）/ `public`（所有人可读），默认 `private`|

**编辑时额外可配置项：**

* 修改名称、描述、图标、可见性

#### 3.1.3 成员管理
**入口：** 知识库右上角「成员」按钮，显示当前成员总数

**成员管理面板（右侧抽屉）：**

|区域|内容|
|-|-|
|成员列表|每个成员显示：头像、用户名、角色标签、加入时间。支持按角色筛选|
|添加成员|搜索框（按用户名/邮箱搜索用户）+ 角色选择 + 「邀请」按钮|
|角色修改|点击成员行右侧按钮，弹出角色选择下拉：Viewer / Editor / Admin|
|移除成员|每个成员行右侧「移除」按钮，Owner 行不显示此按钮；需二次确认|
|搜索用户|输入用户名或邮箱关键字，实时搜索（debounce 300ms），下拉展示匹配结果|

**约束：**

* Owner 不可被移除、不可被修改角色
* Owner 可将知识库所有权转移给另一 Admin（转移后原 Owner 变为 Admin）
* 一个知识库至少有一个成员（Owner）

#### 3.1.4 知识库删除
**入口：** 知识库设置页底部「危险操作」区域

**流程：**

1. 点击「删除知识库」红色按钮
2. 弹出二次确认框，显示知识库名称、文档总数
3. 要求用户输入知识库名称确认
4. 输入匹配后，「确认删除」按钮启用
5. 删除后跳转至知识库列表页

### 3.2 目录系统
#### 3.2.1 功能概述
目录（Section/Notebook）用于组织知识库内的文档，支持**无限层级嵌套**。目录树位于知识库页面左侧面板，是用户导航文档的主要入口。

#### 3.2.2 目录节点类型
|类型|图标|说明|
|-|-|-|
|目录节点|📁|可展开/折叠，包含子目录或文档|
|文档节点|📄|叶子节点，点击进入文档阅读页|

#### 3.2.3 目录树操作
**新建操作：**

|操作|触发方式|行为|
|-|-|-|
|新建根目录|目录树顶部「+ 新建目录」按钮|在目录树根层级创建空白目录，名称默认为「未命名目录」，自动进入内联编辑|
|新建子目录|目录项右键 →「新建子目录」|在该目录下创建空白子目录|
|新建文档|目录项右键 →「新建文档」；或目录树顶部「+ 新建文档」按钮|创建空白文档并跳转到编辑页|

**编辑操作：**

|操作|触发方式|行为|
|-|-|-|
|重命名|双击目录/文档名；或右键 →「重命名」|名称变为可编辑文本框，Enter 确认，Esc 取消|
|拖拽排序|鼠标按住拖拽|可调整同级节点顺序；拖到其他目录上可改变层级|
|移动|右键 →「移动到」|弹出目录选择器，选择目标目录后确认|
|折叠/展开|点击目录节点左侧箭头|折叠或展开子节点；默认全部展开|

**删除操作：**

|操作|触发方式|行为|
|-|-|-|
|删除目录|右键 →「删除」|若目录下有子节点，弹出确认框，提示子节点数量；确认后删除目录及其所有子孙节点|
|删除文档|右键 →「删除」|弹出确认框，确认后删除文档及其关联的卡片、版本历史、评论|

**快捷键（在目录树聚焦时）：**

|快捷键|功能|
|-|-|
|↑ ↓|切换选中节点|
|→|展开选中目录|
|←|折叠选中目录|
|Enter|打开选中文档|
|F2|重命名选中节点|
|Delete|删除选中节点|
|Ctrl+N|在选中目录下新建文档|
|Ctrl+Shift+N|新建目录|
|Ctrl+C|复制选中文档/目录|
|Ctrl+V|粘贴到选中目录下|
|Ctrl+X|剪切（用于移动）|

#### 3.2.4 目录数据排序
每个目录/文档节点存储 `sort_order` 字段（整数）。拖拽操作通过批量更新 `sort_order` 实现。新增节点默认 `sort_order = 同级最大 sort_order + 1000`（留出插入空间）。

### 3.3 文档系统
#### 3.3.1 文档阅读页
**页面布局（三栏）：**

```
┌──────────────────────────────────────────────────────────┐
│  顶部导航栏                                               │
├────────────┬──────────────────────────┬──────────────────┤
│            │                          │                  │
│  目录树    │    文档内容区（主区域）    │   文档大纲       │
│  (左侧面板) │                          │   (右侧面板)     │
│ 宽度 260px │  宽度自适应（最小 540px）   │  宽度 220px     │
│  可折叠    │                          │   可收起         │
│            │                          │                  │
└────────────┴──────────────────────────┴──────────────────┘
```
**文档内容区展示：**

* 文档标题（H1 级别字号，顶部居中或左对齐）
* 标题下方元信息栏：创建人头像+用户名 | 最近编辑人头像+用户名 | 最近编辑时间 | 字数统计
* 分隔线
* 文档正文（渲染后的 HTML）

**右侧大纲面板：**

* 自动提取文档中的 H1、H2、H3 标题
* 层级缩进展示
* 点击标题项平滑滚动到对应位置
* 当前阅读位置的标题高亮

**顶部操作栏（阅读模式）：**

从左到右依次排列：

|按钮|图标/文字|功能|
|-|-|-|
|编辑|✏️ 编辑|进入编辑模式（仅 Editor/Admin/Owner 可见）|
|收藏|⭐|收藏/取消收藏文档|
|评论|💬 评论（N）|展开/收起右侧评论面板，括号内显示评论数|
|分享|🔗 分享|弹出分享面板|
|更多|···|展开更多操作菜单（见下方）|
|导出|📥 导出|展开导出格式选择菜单|

**「更多」菜单项：**

|菜单项|说明|
|-|-|
|复制文档|创建当前文档的完整副本，标题格式「原标题 (副本)」|
|打印|调用浏览器打印功能|
|历史版本|展开历史版本子菜单：查看版本列表 / 对比版本 / 恢复版本|
|页面权限|展开权限子菜单：成员管理 / 权限设置|
|页面视图设置|展开视图子菜单：自适应宽度（开关）|

**导出菜单项：**

|格式|说明|
|-|-|
|Markdown|导出为 .md 文件，保留原始 Markdown 内容|
|Word|导出为 .docx 文件，渲染 HTML 转换为 Word 格式|
|PDF|导出为 .pdf 文件，渲染 HTML 转换为 PDF|

**分享面板：**

弹出右侧抽屉面板，包含以下内容：

|配置项|说明|
|-|-|
|分享开关|开启/关闭分享链接|
|分享链接|只读展示框 + 「复制链接」按钮|
|访问权限|单选：任何人可查看 / 仅知识库成员可查看|
|有效期|单选：永久有效 / 7天 / 30天 / 自定义日期|
|密码保护|可选开关，开启后设置 4-6 位访问密码|

**评论面板：**

右侧抽屉面板，包含：

* 评论输入框（支持 @提及 知识库成员，V1.0 可选实现）
* 评论列表，每条评论显示：头像、用户名、时间、内容
* 评论者本人可删除自己的评论
* Admin/Owner 可删除任何人的评论
* 评论数实时显示在顶部按钮上

#### 3.3.2 文档编辑页
**进入编辑模式：** 点击阅读页顶部「编辑」按钮，或直接在目录树中双击文档名。

**页面布局变化：**

* 右侧大纲面板收起
* 顶部导航栏替换为编辑工具栏
* 目录树面板保持显示（可手动收起）
* 文档内容区切换为编辑器
* 底部状态栏显示保存状态和字数

**编辑顶部工具栏（完整定义）：**

编辑器顶部固定工具栏，从左到右分为以下功能组：

**组1：插入操作**

|按钮|功能|
|-|-|
|插入 ▼|展开插入菜单，包含以下分类：|
|— 文本格式|表格（/bg）、代码块（/dmk）、附件（/fj）、公式（/gs）、图片（/tp）、超链接（/clj）、提及（/tj）|
|— 样式布局|分割线（/fgx）、高亮信息（/glxx）、引用（/yy）、分栏、布局块|
|— 数据图形|流程图（/lct）、思维导图（/swdt）、数据表、图表、Mermaid绘图（/mer）|
|— 功能卡片|拆分卡片（/zcccfkp）、目录卡片（/mlkp）、子文档列表卡（/zwdlbk）|
|— 其他|统计、第三方应用|

**组2：撤销/重做**

|按钮|快捷键|功能|

#### 3.3.3 图片上传
**触发方式：**

* 点击工具栏「插入」→「图片」
* 直接在编辑器中粘贴剪贴板中的图片（Ctrl+V）
* 拖拽图片文件到编辑器区域
* 输入 `![alt](url)` Markdown 语法
* 斜杠命令 `/tp`

**上传流程：**

1. 用户选择/粘贴/拖拽图片后，编辑器内显示上传进度（placeholder 占位 + 百分比）
2. 图片上传至服务器（POST /api/v1/assets/upload）
3. 服务器接收文件，校验类型（允许 jpg/jpeg/png/gif/webp/svg）、大小（≤ 10MB）
4. 存储至文件存储层（本地磁盘或 S3），返回访问 URL
5. 编辑器将图片 URL 插入文档内容
6. 上传失败时，提示错误信息，不阻塞编辑

**图片展示：**

* 编辑模式中：显示为可缩放的行内图片，点击可拖拽调整大小
* 阅读模式中：点击图片可全屏预览（lightbox），支持左右切换

#### 3.3.5 历史版本
**版本生成规则：**

* 每次手动保存（Ctrl+S 或 Cmd+S）生成一条版本记录
* debounce 自动保存不生成版本
* 每个文档最多保留 50 条版本记录，超出后自动删除最旧的版本

**版本列表页（右侧面板）：**

* 按时间倒序展示
* 每条显示：版本号（#1, #2, ...）、保存时间、保存人用户名+头像

**版本预览：**

* 右侧面板切换为预览模式
* 显示该版本的完整文档渲染内容
* 顶部显示版本信息栏：版本号 | 保存时间 | 保存人 | 字数

**版本对比：**

* 在版本列表中选择两个版本（当前版本 + 历史版本）
* 点击「对比」按钮

* 差异处以底色高亮（新增=绿色、删除=红色、修改=黄色）

**版本恢复：**

* 点击某版本的「恢复此版本」按钮
* 弹出确认框，提示「当前未保存的内容将丢失」
* 确认后，文档内容替换为该版本内容
* 生成一条新版本记录（标记为「从版本 #N 恢复」）
* 仅 Editor/Admin/Owner 有此操作权限

### 3.5 卡片系统
#### 3.5.1 设计原则
文档可以按 `---`（水平分隔线）拆分为多个独立卡片。每个卡片是一个独立的内容块，有自己的编辑区域。

**V1.0 范围：** 拆分 + 拖拽排序 + 独立编辑。不做跨文档引用、不做卡片嵌套。

#### 3.5.2 拆分规则
* 编辑器中，光标所在位置，通过以下方式触发拆分：
    * 工具栏「插入」→「在此处拆分卡片」
    * 斜杠命令 `/zcccfkp`

* 拆分以文档中最大的分隔粒度为单位

#### 3.5.3 卡片视图
**卡片视图切换：**

* 文档阅读/编辑页顶部提供视图切换按钮

* 「卡片视图」：卡片独立排列展示

**卡片视图界面：**

* 卡片垂直排列，每张卡片有独立边框和底部阴影
* 卡片左上角显示序号（卡片 #1、卡片 #2 ...）
* 卡片右上角显示操作按钮：编辑、上移、下移、删除
* 支持拖拽卡片调整顺序

**卡片编辑：**

* 点击卡片「编辑」按钮，卡片内容区域切换为编辑模式
* 编辑工具栏（精简版）：加粗、斜体、标题、列表、代码块、图片
* 卡片编辑自动保存（debounce 2 秒）

#### 3.5.4 V1.0 硬性限制
* ❌ 卡片不可跨文档引用
* ❌ 卡片不支持嵌套
* ❌ 卡片不支持独立标签/分类

3. 选择目标目录（默认当前选中目录，或根目录）
4. 选择标题处理方式：

    * 自动：提取第一行 H1 作为文档标题
    * 手动：用户自定义文档标题

5. 点击「导入」，创建文档并跳转到文档页

#### 3.11.2 批量导入
**入口：** 知识库内，目录树顶部或右键菜单「批量导入」

**流程：**

1. 上传 .zip 文件（支持拖拽或点击上传）
2. 系统解析 zip 文件：

    * 按 zip 内的目录结构创建 Section 树
    * 每个 .md 文件转换为一个 Document
    * 文档内的相对路径图片替换为占位符或提示手动上传

3. 显示解析结果预览：

    * 将创建的目录结构以树形展示
    * 标注总文档数、总目录数

4. 用户确认后执行批量创建

## 4. 界面设计
### 4.1 设计原则
* **三栏布局为主**：左侧导航 + 中间内容 + 右侧辅助
* **全响应式设计**：覆盖桌面端（≥1024px）、平板端（768-1024px）、手机端（< 768px），详见 4.2 节
* **浅色主题**：以白色/浅灰为主色调，蓝色为强调色
* **间距一致**：统一 8px 基础栅格系统
* **视觉层级**：通过字号、字重、颜色灰度区分信息层级
* **触摸友好**：手机和平板端所有可点击区域 ≥ 44×44px，支持滑动手势
* **内容优先**：移动端以文档内容为核心，导航和辅助面板按需唤起

#### 4.2.3 通用响应式规则
**导航与面板：**

* 右侧面板默认折叠的断点：< 1024px
* 展开的面板以浮层/抽屉形式覆盖在内容区上方（z-index 提升），点击遮罩层或文档区可关闭

## 6. API 接口设计
### 6.1 通用规范
* **Base URL**：`/api/v1`
* **认证方式**：`Authorization: Bearer <jwt_access_token>`
* **请求格式**：`application/json`（文件上传使用 `multipart/form-data`）
* **响应格式**：

```json
{
  "success": true,
  "data": { ... },
  "error": null
}
```
* **错误响应**：

```json
{
  "success": false,
  "data": null,
  "error": {
    "code": "FORBIDDEN",
    "message": "您没有权限执行此操作"
  }
}
```
* **分页参数**：`?page=1&page_size=20`，响应中包含 `total`、`page`、`page_size`、`total_pages`

## 8. 部署方案
### 8.1 Docker Compose 部署
**docker-compose.yml：**

```yaml
version: '3.9'

services:
  db:
    image: postgres:16-alpine
    restart: unless-stopped
    environment:
      POSTGRES_DB: knowflow
      POSTGRES_USER: knowflow
      POSTGRES_PASSWORD: ${DB_PASSWORD}
    # 不对外暴露数据库端口，仅 Docker 内网可访问
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U knowflow"]
      interval: 5s
      timeout: 3s
      retries: 10

  redis:
    image: redis:7-alpine
    restart: unless-stopped
    volumes:
      - redis_data:/data

  backend:
    build: ./backend
    restart: unless-stopped
    # 不对外暴露，由 frontend nginx 反代 /api/ 访问
    expose:
      - "8000"
    environment:
      DATABASE_URL: postgresql+asyncpg://knowflow:${DB_PASSWORD}@db:5432/knowflow
      REDIS_URL: redis://redis:6379/0
      SECRET_KEY: ${JWT_SECRET}
      SUPER_ADMIN_EMAIL: ${SUPER_ADMIN_EMAIL:-admin@baidu.com}
      CORS_ORIGINS: '["https://knowflow.ihope.cc","http://knowflow.ihope.cc","http://localhost:8192"]'
      PUBLIC_BASE_URL: https://knowflow.ihope.cc
      STORAGE_TYPE: local
      STORAGE_LOCAL_PATH: /data/uploads
      MAX_UPLOAD_SIZE_MB: 100
    depends_on:
      db:
        condition: service_healthy
      redis:
        condition: service_started
    volumes:
      - uploads_data:/data/uploads

  frontend:
    build: ./frontend
    restart: unless-stopped
    ports:
      - "8192:80"
    environment:
      - VITE_API_BASE_URL=/api/v1
      - VITE_PUBLIC_URL=https://knowflow.ihope.cc
    depends_on:
      - backend

volumes:
  pgdata:
  redis_data:
  uploads_data:
```
> **部署说明：** 以上 docker-compose.yml 适配已有 nginx-proxy 反代的场景。KnowFlow 自身不内置 Nginx 容器——frontend 容器对外暴露 `8192` 端口，由用户已有的 nginx-proxy 统一处理 HTTPS/域名/SSL 等。nginx-proxy 配置参考：
```
server {
    listen 443 ssl http2;
    server_name knowflow.ihope.cc;
    # ... SSL 证书配置 ...
    location / {
        proxy_pass http://127.0.0.1:8192;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
    }
}
```
**.env 文件模板：**

```bash
# 数据库密码（必填，建议 16 位以上随机字符串）
DB_PASSWORD=your_secure_db_password_here

# JWT 签名密钥（必填，建议 32 位以上随机字符串）
JWT_SECRET=your_jwt_secret_at_least_32_chars

# 首次注册的 SuperAdmin 账号（必填，部署后首个管理员）
SUPER_ADMIN_EMAIL=admin@baidu.com
SUPER_ADMIN_USERNAME=admin
SUPER_ADMIN_PASSWORD=your_admin_password
```
### 8.2 启动步骤
```bash
# 1. 克隆项目
git clone <repo_url> knowflow && cd knowflow

# 2. 配置环境变量
cp .env.example .env
# 编辑 .env，设置 DB_PASSWORD、JWT_SECRET、SUPER_ADMIN_EMAIL 等

# 3. 启动所有服务
docker-compose up -d

# 4. 初始化数据库（自动运行 alembic migration）
docker-compose exec backend alembic upgrade head

# 5. 创建 SuperAdmin 账号（读取 .env 中的 SUPER_ADMIN_* 配置）
docker-compose exec backend python -m app.seed_admin

# 6. 创建内置模板数据
docker-compose exec backend python -m app.seed_templates

# 6. 配置 nginx-proxy（已有反代服务）
# 在现有 nginx-proxy 中添加 knowflow.ihope.cc 的 server 块，反代到 127.0.0.1:8192
# （参考上方 docker-compose.yml 后的 nginx-proxy 配置参考）

# 7. 访问
# 打开浏览器访问 https://knowflow.ihope.cc
```

|中文全文搜索精度不足|🟡 中|搜索结果不准确|PostgreSQL 配置 zhparser 分词扩展；V1.1 可迁移至 Meilisearch|
|导出格式保真度不足|🟡 中|Word/PDF 导出排版错乱|使用 python-docx + WeasyPrint 成熟库；提供 Markdown 导出作为保底|
|文件存储扩展性|🟢 低|单机磁盘满|抽象 StorageAdapter 接口，切换 S3 只需改配置|
|版本历史存储膨胀|🟢 低|占用过多磁盘|每个文档上限 50 条版本，自动清理；定时任务清理孤儿附件|

### 11.2 产品风险
|风险|等级|影响|应对措施|
|-|-|-|-|
|功能范围蔓延|🟡 中|V1.0 无法按时交付|严格按 PRD 执行，新需求统一进 V1.1/V2.0|
|权限模型复杂导致安全漏洞|🔴 高|数据泄露|权限校验集中在后端中间件层，全覆盖自动化测试|
|用户体验不符合预期|🟡 中|用户流失|参考成熟产品 UX 模式，早期内测收集反馈快速迭代|

### 11.3 约束条件
|约束|说明|
|-|-|
|部署方式|仅支持 Docker 单机部署，不做 Kubernetes 集群支持（V1.0）|
|浏览器兼容|仅支持 Chrome/Firefox/Edge/Safari 最近两个主版本|
|并发上限|单机部署，设计支持 ≤ 50 并发用户|

|数据库|仅支持 PostgreSQL，不做 MySQL/SQLite 兼容|
|认证方式|V1.0 仅支持用户名+密码，不做 SSO/OAuth|
